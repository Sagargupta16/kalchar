# kalchar UI Consistency Plan

> From a multi-agent UI research + audit pass (8 design dimensions + 12 per-page sweeps, all findings adversarially verified). Diagnosis: the design system tokenizes color/radius/space/motion but has ZERO elevation tokens, so 33 raw shadows + 8 ring treatments drift per-component. Fix is a token foundation + mechanical migration, not a rewrite.

## Strategic summary

The frontend reads "pretty bad" for one structural reason: the design system tokenizes color, radius, space, motion and (partly) type, but has ZERO elevation tokens -- so 33 raw Tailwind shadows (sm/md/lg/xl/2xl), 12 black/white opacity rings, 3+ focus-ring patterns, snapped mt-10/sm:mt-14 rhythm, and inline Motion literals are all per-component pixel decisions with no source of truth. The fix is NOT a visual rewrite; it is a token-foundation + mechanical-migration job that evolves the atelier look. Add a warm, ink-tinted, layered elevation scale (shadows on cream must be brown, not black, or they read as cheap gray murk) with a dark-mode strategy that switches depth from drop-shadow to surface-lightening + a light hairline; collapse the 8 ring treatments to one self-flipping hairline token; give type role classes the sizes they're missing; and fluidize the block/gutter rhythm. Every surface then routes through a role map (Card=e1, hover=e2, popover=e3, lightbox chrome=e4, modal=e5) so "raise every card" becomes a one-line edit -- and a grep CI guard keeps drift from returning.

## Risks / guardrails

- Over-shadowing the quiet atelier register: keep per-layer alpha low (0.05-0.10) and resting cards at e1 (hairline lift), NOT e2+. The art is the hero -- a WORK image plate must never carry a resting drop-shadow; give it shadow-hairline only and let the frame define the edge. Only chips-ON-art and genuine floating UI get real elevation.
- Breaking dark mode: do NOT reuse light drop-shadows in dark (they vanish on the 0.14 ground and add murk). Redefine e1..e5 + hairline under :root.dark to inset-highlight + surface-lighten, and pair surfaces with the existing bg-soft/bg-muted ramp (lighter = higher). Verify every elevated surface reads in dark before shipping Phase 1.
- Motion regression: enumerate transition properties -- transition-all over-animates unlisted props and causes jank; list [transform,box-shadow,border-color,...]. When adding StaggerGroup, children must carry variants ONLY (no initial/animate) or all children fire at once (the classic bug). Drop the static will-change:transform on .hero-plate -- it reserves 2 GPU layers for the page life on low-end Android.
- Art-image integrity: only the LIGHTBOX master image flips to object-contain; leave grid thumbnails object-cover (documented user decision in work-filter.tsx:23-25). Never distort or crop the detail view. Keep hero plate radius language consistent (bump grid plate to radius-lg to match hero/detail).
- Format drift: standardize the shadow color on oklch(var(--shadow-ink)/a) reusing the exact --color-ink triplet (0.18 0.022 60) -- do NOT introduce a parallel HSL triplet as one dimension suggested; one hue source, one syntax already used elsewhere in the file.
- Sequencing: land tokens (Phase 0) before any component edits, and do the grep migration as one atomic sweep -- a half-migrated codebase (some e-tokens, some raw shadows) looks worse than either end state. Run the CI guard only AFTER the sweep is complete or it blocks your own PR.
